# 달력 편집 키 바인딩 변경 디자인

**날짜:** 2026-02-15
**상태:** 승인됨

## 개요

달력 화면에서 편집 모드 진입 키를 Enter에서 'e'로 변경하여 Vim/Helix 스타일 키 바인딩 일관성을 유지합니다.

## 목적

- Vim/Helix 스타일 키 바인딩 철학 준수
- 'e' (edit)의 직관적인 니모닉 활용
- 키보드 중앙에서 효율적인 조작 가능

## 변경 내용

### 핵심 변경
- **파일:** `src/main.rs`
- **함수:** `handle_calendar_key` (109번째 줄)
- **변경:** `KeyCode::Enter` → `KeyCode::Char('e')`

### 변경 전 코드
```rust
| KeyCode::Enter => Some(Msg::CalendarSelectDate),
```

### 변경 후 코드
```rust
| KeyCode::Char('e') => Some(Msg::CalendarSelectDate),
```

## 접근 방식

### 선택된 방식: 직접 수정
단일 키 바인딩 변경이므로 `src/main.rs`에서 직접 수정하는 방식을 선택했습니다.

**장점:**
- 간단하고 직관적한 구현
- 즉시 적용 가능
- 현재 프로젝트 규모에 적합
- 추가 의존성이나 복잡도 없음

**고려했으나 채택하지 않은 방식:**
1. **키 바인딩 모듈 분리** - 현재 요구사항 대비 과도한 엔지니어링
2. **설정 파일 기반** - 매우 과도한 엔지니어링, 불필요한 복잡도

## 영향 범위

### 변경 파일
- `src/main.rs` - 단일 라인 변경

### 영향받는 컴포넌트
- `handle_calendar_key` 함수만 영향받음
- `CalendarSelectDate` 메시지 처리 로직은 변경 없음
- 에디터, 모델, 뷰 등 다른 컴포넌트는 영향 없음

### 호환성
- 기존 기능 변경 없음 (키만 변경)
- 메시지 흐름 동일하게 유지
- 데이터 구조 변경 없음
- 에디터 Normal 모드의 'e'(단어 끝 이동)와 충돌하지 않음 (다른 화면)

## 테스트 전략

### 수동 테스트
1. 애플리케이션 실행
2. 달력 화면에서 'e' 키 입력 → 에디터 모드 진입 확인
3. 달력 화면에서 Enter 키 입력 → 반응 없음 확인
4. 에디터 Normal 모드에서 'e' 키 → 단어 끝 이동 정상 작동 확인

### 검증 체크리스트
- [ ] 'e' 키로 편집 모드 진입 작동
- [ ] Enter 키는 달력에서 반응 없음
- [ ] 에디터의 'e' 키 기능 정상 작동
- [ ] 다른 키 바인딩 영향 없음

### 자동 테스트
- 현재 `tests/integration_tests.rs`에 키 입력 테스트 존재
- 필요시 `handle_calendar_key` 함수에 대한 유닛 테스트 추가 가능
- 단순 키 바인딩 변경이므로 수동 테스트로 충분

## 사용자 경험

### 변경 전 동작
- 달력에서 날짜 선택 후 Enter 키로 편집 시작

### 변경 후 동작
- 달력에서 날짜 선택 후 'e' 키로 편집 시작
- Enter 키는 아무 동작도 하지 않음

### 사용성 개선
- Vim/Helix 사용자에게 더 직관적 ('e' = edit)
- 키보드 중앙에서 손을 움직이지 않고 조작 가능
- 일관된 키 바인딩 철학 유지

### 잠재적 혼란
- 기존 사용자가 Enter 키 습관이 있다면 초기에 적응 필요
- 하지만 프로젝트가 Vim/Helix 스타일을 따르므로 예상 가능한 변경
