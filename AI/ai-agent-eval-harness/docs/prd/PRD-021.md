## 배경 및 목적

PRD-017 에서 시나리오/골든셋 데이터를 SQLite 로 일원화했으나, **에이전트 실행 결과(Trajectory + EvaluationResult)** 는 여전히 파일 시스템에 흩어져 있다. 현재는 9~12개 수준이지만 다음과 같은 압력이 누적되고 있다:

1. **검색·필터링 부재** — 웹 UI 의 Trajectories/Reports 탭에서 "최근 실패만", "agent=ppa", "domain=financial" 같은 1차 필터조차 불가능. 파일명이 `trajectory_<task_id>_<ts>.json` 이라 어떤 도메인/시나리오의 결과인지 파일을 열어야 알 수 있다.
2. **라우팅 평가 추적성 단절** — SPEC-020 에서 추가된 `domain_routing_score` 를 시계열로 비교하려면 모든 평가 결과 JSON 을 파싱해야 한다. SQL 한 줄로 끝낼 수 있는 일이다.
3. **회귀 감지 비용** — 현재 `compare` 는 두 파일을 직접 읽는다. "지난 주 결과 vs 이번 빌드" 같은 시간 기반 비교는 외부 스크립트 없이는 불가능.
4. **데스크톱/웹 단일 데이터 소스** — 시나리오/골든셋이 DB 로 통합된 흐름과 일관되도록 결과물도 같은 DB 에 두는 것이 자연스럽다.

다만 **집계 보고서·비교 결과**는 PR/이슈 첨부, git 커밋, GitHub Actions artifact 등 외부 도구와 강하게 결합되어 있고 본질적으로 "동결된 스냅샷" 이므로, 본 PRD 의 범위에서 의도적으로 제외한다(FR-5).

## 대상 사용자
- 웹 UI 에서 평가 결과를 검색·필터링하려는 평가자
- CI/CD 파이프라인에서 회귀 감지를 자동화하려는 개발자
- 데스크톱 앱 사용자(시나리오 관리와 동일한 데이터 소스 기대)

## 요구사항

### 기능 요구사항 (Functional)

- **FR-1**: `Trajectory` DB 저장
  - `TrajectoryLogger::save_trajectory()` 호출 시점에 `trajectories` 테이블에 한 행 INSERT.
  - 컬럼: `task_id`(PK), `task_description`, `agent_name`, `domain`, `scenario_id`, `success`, `total_iterations`, `started_at`, `ended_at`, `steps_json`(BLOB), `final_state_json`(BLOB), `created_at`.
  - `steps` 와 `final_state` 는 JSON TEXT 로 그대로 저장(하이브리드). `task_id` 가 PK 이므로 동일 ID 재기록은 `INSERT OR REPLACE`.

- **FR-2**: `EvaluationResult` DB 저장
  - `TrajectoryLogger::save_evaluation()` 호출 시점에 `evaluations` 테이블에 INSERT.
  - 컬럼: `task_id`(FK → `trajectories.task_id`), `agent_name`, `domain`, `scenario_id`, `success`, `criteria_score`, `tool_sequence_score`, `domain_routing_score`, `overall_score`, `metrics_json`(BLOB), `golden_set_result_json`(BLOB), `created_at`.
  - 동일 `task_id` 는 `INSERT OR REPLACE`.

- **FR-3**: Dual-write
  - 기존 `save_trajectory`/`save_evaluation` 의 파일 쓰기는 **계속 동작**.
  - DB 쓰기와 파일 쓰기 중 어느 한쪽이 실패해도 다른 쪽은 계속 진행 (best-effort, 에러는 stderr 경고).
  - 본 PRD 단계에서는 파일 쓰기 제거 일정을 명시하지 않는다(향후 별도 PRD).

- **FR-4**: 읽기 경로 DB 우선
  - `list_trajectories_impl(dir)` → DB 에 행이 있으면 DB 의 task_id 목록(파일명 형식 `trajectory_<task_id>_<ts>.json` 으로 표현) 반환, 없으면 파일 시스템 폴백.
  - `get_trajectory_impl(dir, name)` → 파일명에서 task_id 추출 → DB 조회, 실패 시 파일 폴백.
  - `list_reports_impl(dir)` / `get_report_impl(dir, name)` 는 **두 종류** 를 모두 다룬다:
    1. 개별 평가 로그(`evaluation_<task_id>_*.json`) → DB 에서 evaluations 조회
    2. 집계 보고서(`evaluation_report_*.json`) → 파일에서만 (FR-5)

- **FR-5**: 집계/비교는 파일 유지
  - `HarnessRunner::save_report()` (집계) 와 `ReportComparator::save_comparison()` (비교) 는 본 PRD 범위 외.
  - 향후 PRD 에서 별도로 다룬다.

- **FR-6**: `compare` 명령 확장
  - 기존: `compare <baseline_file> <current_file>` (유지).
  - 신규 옵션 (CLI 한정, web 은 향후):
    - `compare --baseline-task <task_id> --current-task <task_id>`
    - `compare --agent <name> --since <ISO8601> --until <ISO8601>` — 두 시간 범위의 평균을 비교.
  - DB 저장된 평가 결과를 입력으로 사용. 두 모드는 **상호 배타적**.

- **FR-7**: Backfill 헬퍼
  - `cargo run -- backfill-results --from <dir>` 명령(CLI 서브커맨드 신규).
  - 디렉토리의 모든 `trajectory_*.json`, `evaluation_*.json` 을 파싱해 DB 에 INSERT OR IGNORE.
  - 실패 항목은 stderr 로 경고하고 계속 진행. 종료 코드는 항상 0(부분 성공 허용).

### 비기능 요구사항 (Non-functional)

- **NFR-1**: 마이그레이션 v3 → v4 는 멱등이며, 기존 데이터(시나리오/골든셋)에 영향을 주지 않는다.
- **NFR-2**: DB 쓰기 실패가 에이전트 실행 자체를 중단시키지 않는다(파일 쓰기와 동일 best-effort).
- **NFR-3**: 기존 워크스페이스 테스트(135개) 가 회귀 없이 통과한다.

## 범위 외 (Out of Scope)
- 집계 보고서(`EvaluationReport`) 의 DB 저장
- 비교 결과(`ComparisonResult`) 의 DB 저장
- 파일 출력 제거 (dual-write 만 도입, 단방향 전환은 후속)
- 웹 UI 의 검색·필터·페이지네이션 UI (저장소 준비만, UI 는 후속)

## 성공 기준
- `cargo run -- run` 한 번에 DB 에 trajectory + evaluation 한 행씩 추가됨이 SQL 로 확인 가능
- 기존 파일 출력은 그대로 동작
- `compare --agent ppa --since 2026-01-01T00:00:00Z --until 2026-12-31T23:59:59Z` 가 성공
- `cargo test --workspace` 0 failures
