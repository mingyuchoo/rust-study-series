## 배경 및 목적

현재 SPA (PRD-002 ~ PRD-008) 는 내장 Axum 서버를 기동한 뒤 사용자가 브라우저로 접속해야 한다. 하지만 비개발자 사용자·현장 QA 팀 등은 **네이티브 데스크톱 앱** 형태의 배포를 선호한다. Tauri 2.x 를 사용하면 Rust 백엔드 + 시스템 WebView 조합으로 Linux/macOS/Windows 네이티브 앱을 단일 코드베이스로 생성할 수 있다.

설계 선택: **내장 Axum 서버 + WebView 원격 URL 로드** 방식. 이점은:
- 기존 SPA (`index.html`, `help.html`) 와 8개 엔드포인트를 **한 줄도 수정하지 않고** 재사용
- Tauri IPC (`invoke()`) 로 재작성할 필요 없음
- 웹 배포와 데스크톱 배포가 항상 동일 동작 보장

단점: 로컬호스트 포트를 소비. 사용자가 신뢰하는 환경이므로 허용 가능.

## 대상 사용자
- GUI 앱 설치를 선호하는 QA/PM
- 매번 `cargo run -- serve` + 브라우저 탭을 여는 것이 번거로운 개발자

## 요구사항

### 기능 요구사항
- **FR-1**: `crates/eval-harness/src/lib.rs` 추가 → `pub mod tui; pub mod web; pub mod desktop_helpers;`. 기존 `main.rs` 는 `use eval_harness::{tui, web};` 로 전환. 기존 바이너리 동작/테스트 불변.
- **FR-2**: `desktop_helpers` 모듈에 다음 제공:
  ```rust
  pub fn pick_free_port() -> io::Result<u16>;
  pub fn wait_for_port(host: &str, port: u16, timeout_ms: u64) -> bool;
  ```
- **FR-3**: 리포 루트에 `desktop/` 디렉토리 생성:
  ```
  desktop/
    Cargo.toml         (package, NOT workspace member — exclude)
    tauri.conf.json    (Tauri 2.x 설정, windows: [] 로 동적 생성)
    build.rs           (tauri-build)
    dist/index.html    (플레이스홀더)
    icons/             (플레이스홀더 아이콘)
    src/main.rs        (Tauri 부트스트랩 + 서버 기동)
  ```
  루트 `Cargo.toml` 에 `[workspace] exclude = ["desktop"]` 추가.
- **FR-4**: `desktop/src/main.rs`:
  1. `eval_harness::desktop_helpers::pick_free_port()` 로 포트 확보
  2. 별도 스레드에서 `eval_harness::web::run_server(...)` 기동 (프로젝트 루트 상대 경로 사용)
  3. `wait_for_port(...)` 로 기동 완료 대기
  4. Tauri `setup` 훅에서 `WebviewWindowBuilder::new` + `WebviewUrl::External("http://127.0.0.1:<port>")` 로 윈도우 생성
- **FR-5**: README 섹션에 Tauri 빌드 명령 + 시스템 의존성 표 (Linux `libwebkit2gtk-4.1-dev`, macOS Xcode CLT, Windows WebView2).

### 비기능 요구사항
- **NFR-1**: 워크스페이스(`cargo build`) 가 시스템에 Tauri 의존성이 없어도 계속 성공해야 함 (`desktop/` exclude).
- **NFR-2**: `cargo test -p eval-harness` 는 데스크톱 헬퍼 단위 테스트까지 포함하여 그대로 통과.
- **NFR-3**: 데스크톱 앱 종료 시 내장 서버도 프로세스 종료와 함께 정리 (detached thread 로 충분, OS 프로세스 종료 시 자동 정리).

## 성공 기준
- `cargo test -p eval-harness` 통과 (lib 전환 후 회귀 없음 + SPEC-009 TC 통과)
- `cargo build -p eval-harness` 및 `cargo build --workspace` 성공
- `desktop/` 내부에서 `cargo tauri dev` 를 실행하면(시스템 의존성 충족 시) 데스크톱 윈도우가 열리고 7-탭 SPA 가 동작
- README 의 데스크톱 섹션에 빌드/실행/의존성 설치 명령이 명시됨

## 범위 외
- Native IPC (`invoke()`) 기반 백엔드 호출 — 현재는 HTTP API 재사용
- 앱 번들링·서명·자동 업데이트
- 시스템 트레이·알림 통합
- `desktop/` 크레이트의 CI 빌드 (시스템 의존성 요구로 별도 runner 필요)

## 제약 조건
- `crates/` 디렉토리 구조는 변경 최소화 (새 파일 2개: `lib.rs`, `desktop_helpers.rs`; `main.rs` 수정)
- `desktop/` 은 워크스페이스 밖 standalone 프로젝트 → 기존 CI/워크스페이스 영향 없음
- Tauri 2.x 사용 (stable)
