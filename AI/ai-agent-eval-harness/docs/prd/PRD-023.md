## 배경 및 목적

PRD-022 (Layer 1) 는 *기존에 컴파일된 도구* 의 재조합으로 새 도메인을 만들 수 있게 했다. 그러나 **완전히 새로운 도구 행위**(예: "사용자의 의료 기록 데이터베이스를 조회", "특정 회사 내부 API 호출") 는 여전히 Rust 코드 + recompile 이 필요하다.

PRD-023 은 *generic HTTP delegation* 으로 이 마지막 제약을 푼다:
- 사용자가 자기 Python/Node/Go 서버를 띄우고 endpoint 만 등록
- harness 는 그 endpoint 를 일반 도구처럼 LLM 에게 노출
- 평가 하네스 자체는 변경/recompile 없음
- 도구 작성 언어 무관

WASM/Lua/cdylib 같은 더 강력한 동적 로딩 대안은 복잡도·보안 비용이 훨씬 크고, HTTP delegation 이 *동일한 동적성* 을 훨씬 단순한 모델로 제공한다(평가 harness 라는 단일 사용자 맥락에서).

## 대상 사용자
- 평가 도메인에 특화된 도구를 자기 서비스로 구현하고 싶은 연구자
- 사내 API 와 LLM 에이전트를 빠르게 통합하고 싶은 팀
- Rust 환경 없이 평가 harness 를 확장하고 싶은 비-Rust 개발자

## 요구사항

### 기능 요구사항 (Functional)

- **FR-1**: `external_tools` 테이블 (v6 마이그레이션)
  ```sql
  CREATE TABLE external_tools (
      name           TEXT NOT NULL,
      domain         TEXT NOT NULL REFERENCES domains(name) ON DELETE CASCADE,
      description    TEXT NOT NULL DEFAULT '',
      method         TEXT NOT NULL DEFAULT 'POST',
      url            TEXT NOT NULL,
      headers_json   TEXT,            -- 선택 HTTP 헤더 (JSON 객체)
      body_template  TEXT NOT NULL,   -- {{var}} 치환 템플릿
      params_schema  TEXT NOT NULL,   -- JSON Schema (LLM 노출용)
      timeout_ms     INTEGER NOT NULL DEFAULT 10000,
      created_at     TEXT NOT NULL DEFAULT (datetime('now')),
      PRIMARY KEY (domain, name)
  );
  ```

- **FR-2**: 등록 흐름
  - `domains::register_all` 다음에 `register_external_tools_from_db(&mut registry)` 호출
  - 헬퍼는 `try_installed_store()` 로 전역 store 접근, `list_external_tools()` 결과를 순회하며 각 row → `HttpCallTool::new(...)` → `register_with_domain(arc, &row.domain)`
  - store 가 install 안 된 경우(테스트 등) 조용히 skip

- **FR-3**: `HttpCallTool` 동작
  - construct: `(name, description, method, url, headers, body_template, params_schema, timeout)`
  - `BaseTool::metadata()` → name=`<row.name>`, parameters_schema=`row.params_schema` 그대로
  - `BaseTool::execute(params)`:
    1. `body_template` 안의 `{{key}}` 를 `params[key]` 로 치환 (str/number/bool 직렬화)
    2. `reqwest::blocking::Client` 로 호출, `timeout_ms` 적용
    3. 응답이 200 + valid JSON object → 그대로 HashMap 으로 반환 + `success: true`
    4. 그 외 → `{ "success": false, "error": "..." }`

- **FR-4**: REST CRUD
  ```
  GET    /api/external-tools                     → 모든 external tools (도메인별)
  GET    /api/external-tools/:domain             → 도메인별
  POST   /api/external-tools/:domain             → 신규 (body: ExternalToolUpsert)
  PUT    /api/external-tools/:domain/:name       → 갱신
  DELETE /api/external-tools/:domain/:name       → 삭제
  ```
  CRUD 핸들러는 `agent_core::domain_router::invalidate_cache()` 호출(키워드는 변경 안 되지만, registry 는 다음 `load_all_tools` 호출에서 새 external tool 을 자동 로드).

- **FR-5**: URL allowlist (선택)
  - 환경변수 `EVAL_HARNESS_HTTP_TOOL_ALLOWLIST` 가 콤마 구분 prefix 리스트(예: `https://internal.example.com/,http://localhost:`)
  - 설정되지 않으면 모든 URL 허용 (단일 사용자 가정)
  - 설정된 경우 prefix 매칭 실패 → POST/PUT 400

- **FR-6**: 즉시 노출
  - `PpaAgent::load_all_tools` 가 매 `execute_task` 시작 시 호출되므로(기존 동작), CRUD 후 다음 task 부터 자동 반영
  - 명시적 invalidate 불필요

- **FR-7**: Web UI
  - Domains 탭에서 도메인 선택 시 해당 도메인의 external tools 리스트 + 신규/편집 폼 노출
  - 폼 필드: name, description, method (GET/POST/PUT/DELETE), url, headers (JSON textarea), body_template, params_schema (JSON textarea), timeout_ms

### 비기능 요구사항 (Non-functional)

- **NFR-1**: v5 → v6 마이그레이션은 멱등이며 기존 데이터에 영향 없음
- **NFR-2**: external_tools 가 비어 있을 때 기존 도구만 등록되어 회귀 0
- **NFR-3**: HttpCallTool 호출은 동기 blocking, 단 timeout 으로 PPA 루프 최대 지연 제한
- **NFR-4**: 응답 파싱 실패는 panic 없이 `success: false` 로 격리
