## 배경 및 목적

현재 `agent-core/src/llm_client.rs::create_perceive_prompt` 와 `create_policy_prompt` 는 Perceive/Policy 두 PPA 스테이지의 system/user 프롬프트 4종을 **Rust 소스에 하드코딩** 하고 있다. 도메인이 동적으로 확장(SPEC-022) 되고 외부 HTTP 도구가 런타임에 등록(SPEC-023) 되는 구조가 갖춰졌지만, **도메인별 맥락(예: 의료 도메인의 규제 안내, 금융 도메인의 수치 단위 규약)을 프롬프트에 반영할 방법이 없다**. 또한 프롬프트 수정 시 재컴파일이 필수이며, 수정 전/후 결과를 재현하거나 비교할 수 없다.

PRD-025 는 이 문제를 다음 세 축으로 해결한다:
1. **분리**: 프롬프트를 코드에서 데이터로 옮긴다. 4종 번들을 DB 로 이관.
2. **버전관리**: 불변 버전 + 활성 플래그. 편집 = 새 버전 생성. 활성화/롤백은 1-클릭.
3. **추적성**: 각 실행의 Trajectory 에 사용된 `prompt_set_id` 를 기록 → "이 점수가 어느 프롬프트로 나온 결과인가" 를 DB 단 한 번의 JOIN 으로 답할 수 있다.

"Layer 0" (하드코딩) → "Layer 1" (DB + CRUD + 버전관리) 이동이라는 점에서, 도메인 / 시나리오 / 도구의 동적화 축을 **프롬프트** 로 확장하는 일관된 방향이다.

## 대상 사용자
- 도메인 맞춤 프롬프트 설계를 반복 실험하려는 평가 하네스 사용자
- 프롬프트 변경의 회귀 영향을 정량적으로 비교하려는 연구자
- 코드를 건드리지 않고 도메인 정책(금칙어, 출력 형식 강제 등)을 업데이트하고 싶은 운영자

## 요구사항

### 기능 요구사항 (Functional)

- **FR-1**: `prompt_sets` 테이블 (v7 마이그레이션)
  ```sql
  CREATE TABLE prompt_sets (
      id              INTEGER PRIMARY KEY AUTOINCREMENT,
      domain_name     TEXT NOT NULL REFERENCES domains(name) ON DELETE CASCADE,
      version         INTEGER NOT NULL,
      perceive_system TEXT NOT NULL,
      perceive_user   TEXT NOT NULL,
      policy_system   TEXT NOT NULL,
      policy_user     TEXT NOT NULL,
      notes           TEXT,
      is_active       INTEGER NOT NULL DEFAULT 0,
      is_bootstrap    INTEGER NOT NULL DEFAULT 0,
      created_at      TEXT NOT NULL DEFAULT (datetime('now')),
      UNIQUE (domain_name, version)
  );
  CREATE UNIQUE INDEX idx_prompt_sets_active_per_domain
      ON prompt_sets(domain_name) WHERE is_active = 1;
  ```

- **FR-2**: Bootstrap 시딩
  - `SqliteStore::init_schema()` 에서 v7 마이그레이션 직후 `seed_bootstrap_prompt_sets()` 를 1회 호출
  - 헬퍼는 `domains` 테이블의 모든 이름을 순회하며 해당 도메인에 PromptSet 이 하나도 없으면 v1 을 삽입
  - v1 의 4개 필드는 현재 `llm_client.rs` 의 문자열을 그대로 복사하되, user 프롬프트는 아래 슬롯으로 치환:
    - `perceive_user`: `"작업: {task_description}\n\n환경 상태:\n{environment_state}{context}\n\n위 정보를 분석하여 JSON 형식으로 출력하세요."`
    - `policy_user`: `"작업: {task_description}\n\n인지된 정보:\n{perceived_info}\n\n사용 가능한 도구:\n{tools}{context}\n\n위 도구 중 적절한 것을 선택하고..."`
  - 삽입 행의 `is_active=1`, `is_bootstrap=1`
  - 런타임 행동이 마이그레이션 전과 bit-for-bit 동일해야 한다

- **FR-3**: 런타임 연결
  - `LlmClient::create_perceive_prompt` / `create_policy_prompt` 시그니처에 **`domain: &str`** 매개변수 추가
  - 두 함수는 먼저 `try_installed_store().get_active_prompt_set(domain)` 으로 활성 PromptSet 을 조회
    - `domain` 에 활성 PromptSet 이 없으면 `"general"` 도메인의 활성 PromptSet 을 폴백
    - `general` 도 없으면 컴파일된 디폴트 문자열 상수(bootstrap 시드와 동일 본문) 사용
  - 조회 결과를 렌더링에 사용하고, 반환값과 별도로 `(Vec<Message>, Option<i64>)` 형태로 `prompt_set_id` 도 호출자에게 전달

- **FR-4**: 슬롯 규약
  - 렌더러는 `fn render(tmpl: &str, vars: &HashMap<&str, String>) -> String` 단일 함수
  - 지원 슬롯 이외의 문자열은 그대로 보존
  - system 프롬프트: `{domain_name}` 치환만. 없는 경우 null-effect
  - 리스트 직렬화(예: `{tools}`) 는 호출자가 미리 줄바꿈 구분 문자열로 조립 후 전달

- **FR-5**: 저장 시 검증
  - POST `/api/domains/:name/prompts` 본문 파싱 → 각 필드의 필수 슬롯 존재 확인
  - 누락 슬롯이 하나라도 있으면 HTTP 400 + `{"error": "missing slots", "missing": {"policy_user": ["{tools}"]}}` 형식
  - system 은 필수 슬롯 없음 → 검증 생략

- **FR-6**: REST 엔드포인트
  ```
  GET    /api/domains/:name/prompts                    → 해당 도메인의 전체 버전 목록 (최신이 먼저)
  GET    /api/domains/:name/prompts/:version           → 단일 버전 전체 본문
  POST   /api/domains/:name/prompts                    → 새 버전 생성 (body: {perceive_system, perceive_user, policy_system, policy_user, notes?})
  PUT    /api/domains/:name/prompts/:version/activate  → 해당 버전을 활성화 (트랜잭션 내 기존 활성 해제 포함)
  DELETE /api/domains/:name/prompts/:version           → 삭제
  ```
  POST 응답에 자동 증가된 `version` 반환.

- **FR-7**: 삭제 규칙
  - 활성 버전 삭제 → 409 `{"error": "cannot delete active version"}`
  - bootstrap 버전 삭제 → 409 `{"error": "cannot delete bootstrap version"}`
  - 도메인 CASCADE: 도메인 삭제 시 `ON DELETE CASCADE` 로 해당 도메인의 모든 PromptSet 함께 삭제

- **FR-8**: 궤적 추적
  - `Trajectory` 구조(`agent-models/src/models.rs`)에 `pub prompt_set_id: Option<i64>` 필드 추가
  - `PpaAgent::execute_task` 는 첫 Perceive 호출 시 조회된 PromptSet 의 id 를 Trajectory 에 기록
  - 하나의 실행 내에서 PromptSet 이 변경되는 상황은 없다고 가정(고정값)
  - `trajectories` 테이블과 `evaluations` 테이블에 `prompt_set_id INTEGER NULL` 컬럼 추가 (마이그레이션 v7)
  - GET `/api/trajectories/:name` 응답의 `trajectory` 객체에 `prompt_set_id` 가 포함되어 Web UI 에 표시 가능해야 함

- **FR-9**: Web UI
  - Domains 탭의 도메인 편집기 하단(external tools 섹션 위 또는 아래)에 "Prompts" 서브 섹션 추가
  - 서브 섹션 구성:
    - 좌측: PromptSet 버전 리스트 (`v{N} [active]`, `[bootstrap]` 뱃지 표시)
    - 우측: 선택된 버전의 4-필드 편집기 (`<textarea>` 모노스페이스, 각 textarea 상단에 해당 필드가 지원하는 슬롯 힌트 `{task_description}, ...`)
    - 버튼: "새 버전으로 저장" (POST), "이 버전 활성화" (PUT activate), "삭제" (DELETE), "초기화"
    - 활성/bootstrap 은 편집 불가(readonly) — 편집하면 항상 새 버전으로 저장됨을 UI 로 명시
  - 궤적 패널: 선택된 Trajectory 에 `prompt_set_id` 가 있으면 제목 옆에 `prompt_set #{id} v{version}` 배지 표시 (도메인 CRUD 결과로부터 버전 lookup 1회)

### 비기능 요구사항 (Non-functional)

- **NFR-1**: v6 → v7 마이그레이션은 멱등이며 기존 `external_tools`, `domains`, `trajectories`, `evaluations` 데이터를 보존한다
- **NFR-2**: bootstrap 시드가 이미 존재하는 재기동 시 재-시드 하지 않는다 (domain 당 PromptSet 개수 > 0 이면 skip)
- **NFR-3**: PromptSet 조회는 PPA 스텝마다 수행되므로 O(1) index lookup 이어야 하며 동일 task 내에서 1회 캐시하는 것을 허용한다
- **NFR-4**: 활성 버전 전환은 단일 트랜잭션으로 원자적이어야 한다 (동시 전환 시 index 충돌로 실패하면 재시도는 호출자 책임)
- **NFR-5**: 템플릿 렌더링에서 슬롯 치환 실패(예: 사용자 실수로 `{task_description}` 제거)는 저장 시점 검증에서 이미 거부되므로 런타임 panic 발생 금지

## 성공 기준

1. `curl -X POST /api/domains/customer_service/prompts -d '{...}'` 으로 v2 생성 → `PUT .../v2/activate` → 해당 도메인의 에이전트 실행이 v2 의 본문을 그대로 사용하는 것이 DB 의 새 Trajectory 행에서 `prompt_set_id` 로 확인 가능
2. 기존 테스트 스위트(`cargo test --workspace`) 0 failures. 즉, bootstrap 시드가 기존 하드코딩과 완전 동등
3. Web UI 에서 도메인 선택 → Prompts 섹션 → 편집 → "새 버전으로 저장" → 활성화 → 실행 탭에서 단일 시나리오 실행 → 궤적 탭에 새 버전 배지 표시
4. 활성 버전 DELETE 시 409, bootstrap DELETE 시 409
5. 필수 슬롯 누락된 POST → 400 + 누락 목록 반환

## 범위 외 (Out of Scope)

- 프롬프트 내 LLM 템플릿 고급 기능(조건문, 루프, 함수 호출) — 의도적으로 단순 슬롯만
- 버전 간 Diff UI (Web 단에서 두 버전 비교). 읽기는 지원하나 시각적 diff 는 후속
- 스테이지 독립 버전관리 (perceive 와 policy 를 별도 버전으로) — 번들 확정
- 프롬프트에 대한 RBAC / 승인 워크플로우
- A/B 테스트 기반 자동 승자 선택

## 제약 조건

- 기존 프롬프트 문구는 v1 bootstrap 으로 1:1 이동. 의미 변경 금지
- 추가 의존성 금지 (Handlebars/Tera/MiniJinja 도입 없이 `str::replace` 로 충분)
- `LlmClient::create_perceive_prompt` / `create_policy_prompt` 시그니처 변경은 Rust 빌드에 전파되므로 호출부(`PpaAgent::perceive_step`, `policy_step`) 동시 수정 필요
- `Trajectory` 필드 추가는 Serialize/Deserialize 호환을 위해 `#[serde(default)]` 사용
