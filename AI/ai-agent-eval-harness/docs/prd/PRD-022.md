## 배경 및 목적

PRD-019(시나리오/골든셋 CRUD), SPEC-020(도메인 auto-routing), SPEC-021(궤적/평가 DB 저장)을 거치면서 평가 데이터 전체가 SQLite 단일 소스로 모였다. 그러나 **"도메인" 그 자체는 여전히 Rust 코드와 컴파일 상수에 분산**되어 있다:

- `crates/domains/src/lib.rs::register_all()` — 어떤 도구가 LLM 에 노출되는지(authoritative)
- `crates/agent-core/src/domain_router.rs::DOMAIN_KEYWORDS` — 라우터 키워드(컴파일 상수)
- `domain_tools` 테이블 — 메타데이터 표시용으로만 사용(런타임 무관)

이 분리 때문에 새 도메인을 추가하려면 항상 코드 변경 + recompile 이 필요하다. 평가자가 "healthcare" 같은 새 도메인을 만들어 `read_file`/`write_file` 등 *기존 도구를 재조합* 하는 시나리오/골든셋을 빠르게 추가하지 못한다.

본 PRD 는 **"기존에 컴파일된 도구는 그대로 두고, 도메인이라는 메타 단위만 동적 CRUD"** 를 가능하게 한다. 진짜로 새 *도구 행위* 가 필요한 경우는 후속 PRD-023 (Layer 2 — `HttpCallTool`) 에서 다룬다.

## 대상 사용자
- 새 도메인을 빠르게 실험하고 싶은 평가자/연구자
- 기존 Rust 도구 집합(파일 IO, 분류, 계산 등)을 재조합해 도메인별 시나리오를 분리하고 싶은 팀

## 요구사항

### 기능 요구사항 (Functional)

- **FR-1**: REST 도메인 CRUD
  ```
  GET    /api/domains                       → 모든 도메인 (name, description, scenario_count, tool_class_names[], keywords[])
  GET    /api/domains/:name                 → 단건 상세
  POST   /api/domains                       → 생성 (body: DomainUpsert)
  PUT    /api/domains/:name                 → 갱신 (description, tool_class_names, keywords 교체)
  DELETE /api/domains/:name                 → 삭제 (cascade). 부트스트랩 도메인은 409 Conflict.
  ```
  요청 본문:
  ```json
  {
    "description": "...",
    "tool_class_names": ["read_file", "customer_service__classify_inquiry"],
    "keywords": ["환자", "처방", "치료", ...]
  }
  ```

- **FR-2**: 도구 카탈로그 조회
  - `GET /api/tools/catalog` → `[{name, description, domain, parameters_schema, safety_level}, ...]`
  - 기존 `list_tools_impl()` 결과를 그대로 사용. 추가 필드 없음.

- **FR-3**: 라우터 키워드 DB 이전
  - `agent_core::domain_router::DOMAIN_KEYWORDS` const 제거.
  - `select_domains(task)` 가 첫 호출 시 `try_installed_store()` → `list_all_domain_keywords()` 로 캐시 채움.
  - CRUD 핸들러가 `agent_core::domain_router::invalidate_cache()` 호출.
  - 캐시는 `RwLock<Option<HashMap<String, Vec<String>>>>` 한 개.

- **FR-4**: Cascade 삭제
  - `domain_keywords (domain TEXT REFERENCES domains(name) ON DELETE CASCADE)` FK 추가.
  - `domain_tools` 는 이미 cascade 있음(SPEC-017 v3 스키마).
  - `eval_scenarios` 도 cascade 있음.

- **FR-5**: 부트스트랩 보호
  - `domains::known_domains()` 에 등록된 이름은 `delete_domain` 에서 거부.
  - 갱신·키워드 변경은 허용 (사용자가 키워드를 튜닝할 수 있어야 함).

- **FR-6**: Web UI
  - Manage 탭 안의 sub-section 또는 Domains 탭 신규.
  - 폼: 이름(생성 시만), 설명(textarea), 도구 multi-select(catalog API 결과), 키워드(comma-separated 또는 chip).
  - 부트스트랩 도메인은 "삭제 불가" 배지 표시.

- **FR-7**: v5 마이그레이션 시드
  - `migrate_v5_domain_keywords()` 가 새 테이블을 만들고, financial/customer_service 의 기존 const 키워드 목록을 `INSERT OR IGNORE` 로 적재.
  - 멱등이며, 사용자가 키워드를 수정한 뒤 재기동해도 덮어쓰지 않음.

### 비기능 요구사항 (Non-functional)

- **NFR-1**: v4 → v5 마이그레이션은 멱등이며 기존 데이터에 영향이 없다.
- **NFR-2**: 라우터 캐시 갱신 latency: CRUD 호출 후 다음 `select_domains` 호출에서 즉시 반영.
- **NFR-3**: 기존 워크스페이스 테스트(143개) 가 회귀 없이 통과한다.

## 범위 외 (Out of Scope)
- 새로운 *도구 행위* 의 동적 추가 (Layer 2, PRD-023)
- 도메인 단위 RBAC/권한 (단일 사용자 환경 가정)
- 도메인 export/import (수동 SQL 또는 후속)

## 성공 기준
- `curl -X POST /api/domains -d '{"description":"의료","tool_class_names":["read_file","write_file"],"keywords":["환자","처방"]}'` 으로 새 도메인 생성 → `GET /api/domains` 응답에 포함
- 같은 도메인의 시나리오/골든셋을 SPEC-019 CRUD 로 추가 가능
- task_description 에 "환자 의무기록을 분류해줘" 가 들어있을 때 SPEC-020 라우터가 새 도메인을 top-1 으로 선택
