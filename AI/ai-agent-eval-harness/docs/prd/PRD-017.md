## 배경 및 목적
현재 `eval_data/` 하위에 평가 시나리오(YAML, 도메인별 1 파일) 와 골든셋(JSON, 도메인별 1 파일) 이 각각 파일로 관리되고 있다. 파일 수는 작지만 다음과 같은 한계가 있다:

1. **쿼리 불가**: 특정 도구를 사용하는 시나리오만 찾거나, 난이도별로 집계하는 등 조건 기반 조회가 불가능하다.
2. **원자적 업데이트 부재**: 여러 도메인의 시나리오를 동시에 갱신할 때 트랜잭션이 없다.
3. **확장성 제한**: 수백~수천 개의 시나리오로 증가할 경우 디렉토리 스캔 + 전체 파일 파싱이 부담이 된다.
4. **데스크톱/웹 공유**: 여러 진입점이 같은 데이터셋을 공유할 때 일관성 보장이 어렵다.

SQLite 로 전환하면 단일 파일로 배포 용이성을 유지하면서도 표준 SQL 로 조회/관리가 가능해지고, 추후 LLM Judge 결과·실행 히스토리 등을 같은 DB 에 수용할 확장 기반이 생긴다.

YAML/JSON 파일은 **seed 소스**로 유지하여 (a) 깃 diff 로 데이터 변경을 리뷰할 수 있고, (b) 새 체크아웃/새 환경에서 DB 를 자동 복원할 수 있게 한다.

## 대상 사용자
- 평가 데이터셋을 쿼리하고 싶은 평가자/연구자
- 다중 진입점(CLI/TUI/Web/Desktop) 을 사용하는 개발자
- 추후 대규모 평가 데이터셋(수백 이상)을 운영할 팀

## 요구사항

### 기능 요구사항 (Functional)
- **FR-1**: `eval_data/eval_harness.db` (기본 경로) SQLite DB 를 단일 소스로 사용한다. 테이블은 최소 `domains`, `domain_tools`, `eval_scenarios`, `golden_sets`, `schema_migrations` 로 구성한다.
- **FR-2**: DB 가 없거나 비어 있으면 기동 시 `scenarios_dir`/`golden_sets_dir` 의 YAML/JSON 을 읽어 자동 seed 한다. seed 는 idempotent 해야 하며(이미 있으면 skip), 파일은 삭제하지 않는다.
- **FR-3**: `ScenarioLoader::load_all_domains(dir)` 는 내부적으로 DB 를 조회하도록 재구현된다. 인자 `dir` 는 seed 소스 힌트로만 사용되거나 deprecated 대상이 된다(하위 호환을 위해 시그니처는 유지). `load_golden_sets(path)`·`load_all_golden_sets(dir)` 도 동일.
- **FR-4**: `eval-harness.toml`:
  ```toml
  [data]
  db_path         = "eval_data/eval_harness.db"   # 신규
  scenarios_dir   = "eval_data/eval_scenarios"    # seed 소스 (옵셔널)
  golden_sets_dir = "eval_data/golden_sets"       # seed 소스 (옵셔널)
  ```
  `db_path` 는 절대/상대 모두 허용하며, 상대 경로는 설정 파일 디렉토리 기준으로 해석(기존 `resolve_relative` 규칙 재사용).
- **FR-5**: 모든 진입점은 `EvalDataStore::open_and_seed(&paths).await?` 같은 단일 API 로 DB 를 초기화한다. 초기화 실패는 종료 코드 1 로 명확히 보고한다.
- **FR-6**: `sqlx` = { features = ["sqlite", "runtime-tokio-rustls", "macros"] }; 기존 워크스페이스에 `tokio` 는 이미 있으므로 그대로 사용. 동기 진입점은 `tokio::runtime::Runtime::block_on` 으로 브릿지한다.
- **FR-7**: 동적 객체 필드는 `TEXT` 컬럼에 JSON 직렬화하여 저장한다. 역직렬화 실패는 seed 시점 검증 오류로 보고한다.

### 비기능 요구사항 (Non-Functional)
- **NFR-1**: 기존 CLI/웹 사용자가 별도 설정 없이도 동작해야 한다(파일 seed → DB 자동 생성 → 기존 UX 보존).
- **NFR-2**: DB seed/조회 경로가 기존 파일 로딩 대비 체감 가능한 지연을 만들지 않는다(수십 시나리오 기준).
- **NFR-3**: 테스트는 `tempfile::tempdir` + in-memory 또는 임시 파일 DB 로 격리되어야 하며, 워크스페이스 전역 상태에 의존하지 않는다.
- **NFR-4**: `sqlx` 의 컴파일 타임 쿼리 검증(`query!` 매크로)은 선택 사항이며, 본 PRD 에서는 런타임 검증(`query_as::<_, Row>`) 만 사용하여 빌드 환경에 DB 를 요구하지 않는다.

## 성공 기준
- `cargo test -p data-scenarios` 통과 — seed, 조회, 재기동 시 중복 적재 없음 검증.
- `cargo run -- list` 가 기존 YAML 파일이 있는 상태에서 최초 실행 시 DB 를 자동 생성하고 시나리오 목록을 출력한다.
- 두 번째 실행 시 DB 재사용(파일 변경 없음) 확인.
- 웹/TUI 에서도 동일한 데이터가 보인다.
- 추적성 매트릭스에 PRD-017/SPEC-017 등록.

## 범위 외 (Out of Scope)
- DB 에 실행 히스토리/리포트/LLM Judge 결과 저장 (별도 PRD).
- DB 스키마 마이그레이션 도구 체계(본 PRD 는 initial schema + 간단한 version 테이블만).
- 파일 → DB 의 **양방향 동기화**. seed 는 파일 → DB 단방향만 지원.
- DB 파일 삭제/재빌드 CLI 명령. (수동으로 DB 파일 지우면 다음 기동 시 재 seed.)
- 웹 UI 상에서 시나리오 CRUD.

## 제약 조건
- 기존 `ScenarioLoader::new()` 호출부(`main.rs`, `web/*.rs`, `execution/runner.rs` 등 7 곳)는 시그니처를 가능한 한 유지하여 호출부 수정 최소화.
- `sqlx` 는 `data-scenarios` 크레이트 내부에만 의존으로 두고, 상위 크레이트(eval-harness, execution 등)에는 노출하지 않는다.
- YAML/JSON 원본 파일은 레포지토리에 유지하며 CI 에서 삭제되지 않아야 한다.
- 빌드 시 DB 파일이 필요하지 않아야 한다(`sqlx` offline 모드/매크로 미사용).
