## 배경 및 목적
`desktop/` 경로에는 이미 Tauri 2 기반 데스크톱 래퍼(`eval-harness-desktop`) 와 `tauri.conf.json` 이 존재하지만, 배포용 번들을 생성하려면 개발자가 수동으로 `cd desktop && cargo tauri build --target ...` 명령을 조합해야 한다. 릴리즈 절차가 문서/스크립트로 고정되지 않으면:

- 플랫폼별 번들 포맷(`msi`, `nsis`, `deb`, `appimage`, `dmg`, `app`) 선택이 휘발성으로 남는다.
- CI 또는 다른 기여자가 동일한 산출물을 재현하기 어렵다.
- 기존 빌드 스크립트(`Makefile.toml`) 와 릴리즈 절차의 진입점이 분리된다.

이를 해결하기 위해 기존 `Makefile.toml` 에 플랫폼별 데스크톱 릴리즈 태스크를 추가해 **"하나의 진입점에서 모든 빌드 명령을 실행"** 하는 기존 스타일을 유지한다.

## 요구사항

### 기능 요구사항
- **FR-1**: `Makefile.toml` 에 다음 3개의 플랫폼 태스크를 추가한다.
  - `desktop-release-windows` — Windows MSI/NSIS 번들 생성 (`--target x86_64-pc-windows-msvc`)
  - `desktop-release-linux` — Linux AppImage/Deb 번들 생성 (`--target x86_64-unknown-linux-gnu`)
  - `desktop-release-macos` — macOS `.app`/`.dmg` 번들 생성 (`--target universal-apple-darwin`)
- **FR-2**: `desktop-release-all` 태스크가 존재하며, 내부적으로 `cargo make` 의 `run_task` 플로우 컨트롤로 3개 플랫폼 태스크를 의존성으로 연결한다. (호스트에서 빌드 불가능한 타겟은 `cargo tauri` 가 에러를 반환하지만, 명령 존재 자체는 보장한다.)
- **FR-3**: 각 태스크는
  - `workspace = false` 로 설정 (루트 가상 워크스페이스가 아닌 `desktop/` 크레이트에서 실행)
  - `cwd = "desktop"` 으로 진입
  - `install_crate = { crate_name = "tauri-cli", binary = "cargo", test_arg = ["tauri", "--version"] }` 로 `cargo-tauri` 를 자동 설치
  - 명시적 `--target <triple>` 인자와 `--bundles <format>` 인자로 재현성 확보

### 비기능 요구사항
- **NFR-1**: 기존 태스크(`build`, `release`, `run`, `watch-run`, `test`, `format`, `clippy`) 의 동작/이름에는 영향이 없다.
- **NFR-2**: TOML 문법 오류가 없어야 한다 (`cargo make --list-all-steps` 가 성공).

## 성공 기준
- `Makefile.toml` 에 `desktop-release-windows`, `desktop-release-linux`, `desktop-release-macos`, `desktop-release-all` 4개 태스크가 존재.
- 각 태스크가 `cargo` 와 `tauri` 서브커맨드, 올바른 `--target` triple 을 포함.
- 단위 테스트: `Makefile.toml` 을 읽어 위 태스크 이름/타겟이 문자열로 존재하는지 검증.
