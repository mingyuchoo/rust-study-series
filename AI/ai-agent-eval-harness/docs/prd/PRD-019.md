## 배경 및 목적
PRD-017 에서 `eval_data` 저장소를 SQLite 로 전환하면서 "웹 UI 상에서 시나리오 CRUD" 는 명시적으로 범위 외였다. 이제 저장소 단일화가 완료된 상태에서, 다음과 같은 사용자 시나리오를 만족시키기 위해 CRUD 를 공식 기능으로 추가한다:

1. 평가자가 YAML/JSON 을 직접 편집하지 않고 웹 UI 에서 새로운 시나리오·골든셋을 빠르게 추가할 수 있어야 한다.
2. 잘못 입력한 시나리오를 수정/삭제하는 데 텍스트 에디터 + 재기동 사이클이 필요하지 않아야 한다.
3. 데스크톱/웹 UI 에서 평가 데이터셋을 "한 곳"에서 관리하고 싶다는 요구.

PRD-017 의 "파일 → DB 단방향 seed" 원칙은 **DB 가 비어 있을 때의 초기 적재** 에만 적용된다. 본 PRD 에서는 seed 이후 모든 쓰기는 **DB 단독**에 적용되며, 파일로 역동기화하지 않는다. 기존 YAML/JSON 파일은 레포지토리에 남아 있는 "초기 시드" 로서만 의미를 가진다.

## 대상 사용자
- 웹 UI 로 평가 데이터셋을 관리하려는 평가자/연구자
- 시나리오를 자주 추가·수정하는 개발자
- YAML/JSON 수작업 편집을 피하고 싶은 팀원

## 요구사항

### 기능 요구사항 (Functional)

- **FR-1**: 평가 시나리오 REST CRUD
  ```
  GET    /api/eval-scenarios                     → 전체 목록 (도메인별 그룹)
  GET    /api/eval-scenarios/:domain/:id         → 단건 조회 (기존 /api/scenarios/:domain/:id 병행 유지)
  POST   /api/eval-scenarios                     → 생성 (body: EvalScenarioUpsert)
  PUT    /api/eval-scenarios/:domain/:id         → 수정 (전체 교체)
  DELETE /api/eval-scenarios/:domain/:id         → 삭제
  ```
  요청/응답 본문은 JSON. 시나리오의 `initial_environment`/`success_criteria` 는 임의의 JSON 객체를 허용한다.

- **FR-2**: 골든셋 엔트리 REST CRUD
  ```
  GET    /api/golden-sets/:domain                          → 도메인 골든셋 파일 전체 (기존)
  GET    /api/golden-sets/:domain/:scenario_id             → 단건 조회 (기존)
  POST   /api/golden-sets/:domain                          → 엔트리 생성 (body: GoldenEntryUpsert)
  PUT    /api/golden-sets/:domain/:scenario_id             → 엔트리 수정
  DELETE /api/golden-sets/:domain/:scenario_id             → 엔트리 삭제
  ```

- **FR-3**: 모든 쓰기는 `SqliteStore` 의 트랜잭션으로 `eval_data/eval_harness.db` 에만 반영된다. 어떠한 경우에도 `eval_data/eval_scenarios/*.yaml`, `eval_data/golden_sets/*.json` 파일을 수정·재작성하지 않는다. 시드 소스 파일은 DB 가 존재하지 않거나 비어 있을 때만 `seed_from_fs` 가 읽는다.

- **FR-4**: `eval_scenarios` 테이블의 외래키를 활용한 cascade 삭제. SPEC-017 의 `golden_sets` 에는 FK 가 없었으므로, 본 PRD 에서 마이그레이션으로 `(domain, scenario_id) → eval_scenarios(domain, id) ON DELETE CASCADE` 를 추가한다. 기존 DB 에도 적용 가능하도록 `schema_migrations` 에 새 버전을 기록한다.

- **FR-5**: 에러 매핑
  | 상황 | HTTP | 본문 |
  |------|------|------|
  | 생성 시 동일 `(domain, id)` 존재 | 409 | `{"error":"conflict","detail":"..."}` |
  | 수정/삭제 대상이 없음 | 404 | `{"error":"not_found","detail":"..."}` |
  | 필수 필드 누락, JSON 파싱 실패, 타입 불일치 | 400 | `{"error":"bad_request","detail":"..."}` |
  | DB 오류 등 내부 실패 | 500 | `{"error":"internal","detail":"..."}` |

- **FR-6**: 웹 UI (`crates/eval-harness/src/web/index.html`) 에 "관리(Manage)" 탭을 새 최상위 탭으로 추가한다. 하위 구성:
  - 좌측: 도메인 목록 → 선택 시 해당 도메인의 시나리오/골든셋 테이블 표시.
  - 우측: 선택된 엔트리의 편집 패널 (JSON 텍스트에리어 + "저장/삭제" 버튼).
  - 상단: "새 시나리오", "새 골든셋" 버튼.
  - 현재 정책: 낙관적 업데이트 없이, 쓰기 후 해당 목록을 재조회.

- **FR-7**: 일관성
  - 쓰기 API 는 `SqliteStore` 의 동일 `SqlitePool` 위에서 실행되며, 직후의 읽기 API 가 같은 풀을 사용하므로 read-after-write 가 보장된다.
  - 로더 캐시(`ScenarioLoader` 내부에 캐시가 있다면)는 쓰기 직후 무효화되어야 한다. 본 PRD 에서는 캐시 없음을 전제로 하며, 캐시 도입 시 별도 PRD 에서 처리한다.

### 비기능 요구사항 (Non-Functional)

- **NFR-1**: 변경 전 UX 회귀 없음. 기존 `GET /api/scenarios`, `GET /api/golden-sets`, 실행·비교·궤적 탭은 동일하게 동작한다.
- **NFR-2**: 모든 CRUD 핸들러는 `tokio::test` + in-memory SQLite (`sqlite::memory:`) 단위 테스트로 검증 가능해야 한다 (웹 핸들러는 axum `Router` 를 `tower::ServiceExt::oneshot` 으로 호출).
- **NFR-3**: 입력 검증(필수 필드, ID 규칙 `^[a-z0-9_\-]+$`)은 핸들러 진입 직후에 수행하여 DB 오류로 떨어지지 않게 한다.
- **NFR-4**: 스키마 마이그레이션(`v2: golden_sets FK`)은 기존 DB 에 대해 **파괴적이지 않게** 수행 (새 테이블 생성 → 데이터 복사 → RENAME). 마이그레이션 실패 시 기동을 중단한다.
- **NFR-5**: 로컬 단일 사용자 도구이므로 인증/인가는 범위 외이나, 파괴적 동작(DELETE)은 HTTP 메서드로 명확히 구분한다.

## 성공 기준
- `cargo test -p data-scenarios -p eval-harness` 에서 신규 CRUD 테스트(TC-1 ~ TC-12) 통과.
- 웹 UI 관리 탭에서 시나리오 생성 → 목록 새로고침 → 상세 조회 → 수정 → 삭제 플로우가 수동 E2E 로 동작.
- 시나리오 삭제 후 연결된 골든셋 엔트리가 cascade 로 함께 사라진다.
- 서버 재기동 후에도 변경 내용이 유지된다(파일 seed 로 되돌아가지 않는다).
- `eval_data/eval_scenarios/*.yaml` 및 `eval_data/golden_sets/*.json` 파일의 mtime 이 CRUD 과정에서 변하지 않는다.
- 추적성 매트릭스에 PRD-019/SPEC-019 등록.

## 범위 외 (Out of Scope)
- 도메인(domain) 자체의 생성/삭제 (기존 2개 도메인을 기준으로 동작).
- 변경 이력(audit log)/undo/soft-delete.
- 권한 관리, 사용자 인증.
- 파일로의 역동기화(export) 또는 "DB → YAML" 덤프 CLI.
- 동시성 제어를 위한 낙관적 잠금(ETag/`If-Match`).
- Bulk import/upload (CSV, YAML 업로드).
- 시나리오 복제(Clone) 기능.

## 제약 조건
- SPEC-017 에서 도입한 `SqliteStore` 와 `ScenarioLoader` 브릿지를 유지한다. `ScenarioLoader` 의 기존 시그니처는 변경하지 않는다.
- PRD-017 의 FR-2(seed 소스 파일 유지)는 그대로 준수한다. 파일은 삭제/수정되지 않는다.
- 웹 UI 는 현재의 Vanilla JS/HTML 단일 파일 구조를 유지한다(프레임워크 도입 금지).
- REST 경로는 하이픈 케이스(`eval-scenarios`, `golden-sets`)를 사용하며, 기존 `GET /api/scenarios`, `GET /api/golden-sets` 와 **공존**한다.
